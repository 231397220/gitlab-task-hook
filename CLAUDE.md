# CLAUDE.md

本文件为 Claude Code（claude.ai/code）在此代码仓库中工作时提供指导。

以下行为准则旨在减少 LLM 编码中的常见错误，可与项目专属指令合并使用。

**权衡取舍：** 这些准则偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 先思考，再编码

**不要假设。不要隐藏困惑。主动暴露取舍。**

开始实现之前：
- 明确说明你的假设。如有不确定，先提问。
- 如果存在多种解读，逐一列出，不要默默选择其中一种。
- 如果存在更简单的方案，说出来。必要时可以反驳需求。
- 如果有不清楚的地方，停下来。指出困惑所在。提问。

## 2. 简洁优先

**用最少的代码解决问题。不做推测性开发。**

- 不添加超出需求的功能。
- 单次使用的代码不做抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不对不可能发生的场景做错误处理。
- 如果写了 200 行但 50 行就能搞定，重写它。

问自己："一个高级工程师会说这过于复杂吗？"如果是，简化它。

## 3. 外科手术式修改

**只动必须动的地方。只清理自己制造的混乱。**

修改已有代码时：
- 不要"顺手改进"相邻代码、注释或格式。
- 不要重构没有问题的代码。
- 保持已有风格，即使你会有不同做法。
- 如果发现不相关的死代码，指出来——不要删除。

当你的改动产生孤儿代码时：
- 删除**由你的改动**导致的无用 import、变量、函数。
- 不要删除原本已存在的死代码，除非被明确要求。

检验标准：每一行改动都应能直接追溯到用户的需求。

## 4. 目标驱动执行

**定义成功标准。循环验证直到达成。**

将任务转化为可验证的目标：
- "添加校验" → "为无效输入编写测试，然后让测试通过"
- "修复 bug" → "编写能复现 bug 的测试，然后让测试通过"
- "重构 X" → "确保重构前后测试均通过"

对于多步骤任务，先陈述简要计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

明确的成功标准让你能独立循环推进；模糊的标准（"让它跑起来"）则需要不断澄清。
